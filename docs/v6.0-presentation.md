### P1: 仓库全景

| 类别 | 数量 | 占比 | 代表 |

### P3: 当前痛点

```
zlib/                               # 典型补丁仓现状
├── zlib-optimizs-v1.2.12.patch     # 无 DEP-3 头，不知道谁写的、什么状态
├── zlib-1.2.12.tar.gz              # 源码 tarball 放在仓库里
├── build.sh                        # 构建脚本随意放
├── README.md                       # 唯一的文档，安装方式靠口口相传
└── LICENSE
```

核心问题：
- **无上游追溯**：不知道 patch 对应哪个 upstream commit
- **无状态管理**：不知道 patch 合入上游了没有
- **无依赖/冲突表达**：同名 patch（0001-xxx）靠命名约定暗示互斥，无结构化校验
- **无自动化**：apply 靠手敲 git apply，顺序靠经验
- **仓库间不一致**：有的用 src/ 目录，有的根目录平铺，有的放 tarball

## P4-P6：openEuler 的 patch 应用情况

### P4: openEuler 双机制并存

openEuler 有两套 patch 管理体系，普通包和内核包走不同路径：

| | 普通包（redis 等） | 内核包 |
|------|------|------|
| 驱动文件 | `.spec` + `Patch0000:` | `series.conf` + `guards` (SUSE-origin) |
| 应用方式 | `rpmbuild` → `%autopatch` | `apply-patches` shell + `guards` Perl |
| patch 放置 | 仓库根目录扁平 | `patches.kernel.org/` 目录 + `patches.addon/` |
| 版本管理 | 每个版本一个 fork 分支 | `series.conf` 行内 guard 语法 `+symbol` / `-symbol` |
| Upstream-Status | **无** | **无** |

> 结论：openEuler 自己也未统一，普通包和内核包走两套完全不同的系统。

### P5: openEuler 缺失的能力

从 [docs/research/openEuler-patch-mgmt.md] 调研：

- ❌ **无 Upstream-Status 字段** — 状态散在 changelog + 注释里
- ❌ **无 per-package patch dashboard** — "行业空白"
- ❌ **无强 schema 约束** — patch header 无规范，lint 不存在
- ❌ **双机制并存是历史包袱** — 普通包 RPM 风格 + 内核包 SUSE 风格
- ✓ **条件包含**：guards 的 `+symbol` / `-symbol` 思路与 Kconfig `depends` 同源
- ✓ **构建体系完整**：rpmbuild 全链自动化

### P6: 与业界对比

| 维度 | openEuler | Debian | Buildroot | OpenWrt | **v6.0** |
|------|:--:|:--:|:--:|:--:|:--:|
| patch 顺序 | spec 行序 / series | series 文件 | 目录遍历 | PATCHFILES 列表 | **目录遍历** |
| 元数据 | 无强约束 | DEP-3 | .mk 变量 | Makefile 注释 | **DEP-3 + manifest** |
| 依赖表达 | guards +symbol | Depends: | DEPENDENCIES | DEPENDS | **depends DFS** |
| 冲突表达 | 无 | Conflicts: | depends on ! | 无 | **conflicts** |
| 构建集成 | rpmbuild | dpkg-buildpackage | make | make | **install.build** |
| 一个文件管所有 | ✗ | ✗ | ✓ .mk | ✓ Makefile | **✓ manifest** |

## P7-P10：v6.0 整体方案

### P7: 设计理念

```
Buildroot package/redis/          →  v6.0 src/Redis-7.0.15/
├── redis.mk (版本+编译)          →  ├── manifest.yaml
├── 0001-xxx.patch                →  ├── kunpeng-dtoe/
├── 0002-xxx.patch                →  ├── kunpeng-iouring/
└── Config.in (依赖/冲突)         →  └── rdb-aof-fallback/
```

**核心原则**：文件系统即配置。目录里有什么就打什么，不需要维护 patch 列表。

**融合 5 家业界方案**：

| 方案 | 吸收了什么 | 放弃了什么 |
|------|------|------|
| Buildroot | 目录遍历 apply、.mk 式单一配置文件 | 完整的包构建体系 |
| Kconfig | depends DFS + 环检测 + conflicts | select 反向语义 |
| DEP-3 | patch 头 6 必填 + 条件必填 | series 文件 |
| OpenWrt | --features 子集选择 | CONFIG_ 命名空间 |
| RPM/openEuler | install: configure/build 结构化声明 | .spec 的 Patch0000 显式列表 |

### P8: 目录结构 + Schema

```
boostkit-redis-v6/
├── README.md
├── tools/
│   ├── apply_patch.sh       # 一条命令：clone → apply → build
│   ├── verify.sh            # lint + apply 验证
│   └── lint.py              # manifest + DEP-3 校验
├── .github/workflows/ci.yml
├── docs/schemas.md
└── src/
    └── Redis-7.0.15/
        ├── manifest.yaml    # ★ 3 必填字段 + 可选 depends/conflicts/install
        ├── kunpeng-dtoe/    # feature = 含 .patch 的目录
        ├── kunpeng-iouring/
        ├── rdb-aof-fallback/
        ├── deps/            # 构建依赖（有则放）
        └── rpm_build/       # RPM 打包
```

### P9: 关键字段对比（v5.2 → v6.0）

| 维度 | v5.2 | v6.0 | 降幅 |
|------|:--:|:--:|:--:|
| YAML 文件数 | 2 (upstream.yaml + features.yaml) | 1 (manifest.yaml) | -50% |
| YAML 字段 | ~16 | 3 必填 | -81% |
| 目录层数 | 3 (patches/features/name/) | 1 (name/) | -66% |
| 新增 patch | cp + 写头 + 改 YAML 列表 | cp + 写头 | -33% |
| CI 步骤 | 3 | 1 | -67% |
| 工具脚本 | 4 | 3 | -25% |

### P10: 一条命令全自动

```bash
# 全量 apply（冲突 feature 自动逐个处理）
bash tools/verify.sh

# 选择 feature 子集
bash tools/apply_patch.sh --features "kunpeng-iouring" src/Redis-7.0.15 /tmp/build

# CI: lint gate（秒级）
python3 tools/lint.py all src/*/
```

输出示例：
```
=== boostkit verify ===
--- lint ---
  ✓ 4 manifests, 7 patches, 0 errors
--- manifest + apply ---
  ✓ Redis-7.0.15: redis/redis @ 7.0.15
    → 单独 apply: kunpeng-dtoe        ✓
    → 单独 apply: kunpeng-iouring     ✓
    → 单独 apply: rdb-aof-fallback    ✓
✓ verify 全部通过
```

## P11-P13：方案详情

### P11: 三个关键脚本

| 脚本 | 功能 | 业界对齐 |
|------|------|------|
| `apply_patch.sh` | 读 manifest → 扫 feature 目录 → depends DFS 排序 → conflicts 检查 → clone → apply → install | Buildroot apply-patches.sh + Kconfig depends |
| `verify.sh` | lint gate + clean apply（冲突 feature 逐个验证） | Buildroot check-package |
| `lint.py` | patch 头 6 必填 + 条件必填 + manifest 字段 + depends 环检测 + conflicts 引用 + 孤儿检查 | DEP-3 + Yocto Upstream-Status |

### P12: Schema 核心字段

**manifest.yaml（3+ 可选）**：

```yaml
repo: https://github.com/redis/redis     # 必填
version: 7.0.15                           # 必填
commit: f35f36a265403c...                 # 必填，40-char SHA

features:                                 # 可选：仅当有依赖/冲突时
  kunpeng-dtoe:
    conflicts: [kunpeng-iouring]
  rdb-aof-fallback:
    depends: [kunpeng-dtoe]

install:                                  # 可选：仅当需要编译验证时
  configure: make distclean
  build: make -j$(nproc) USE_KRAIO=1
  deps: [rpm_build/include/kraio.h]
```

**DEP-3 头（9 必填 + 3 条件必填）**：

```
From / Subject / Description / Origin / Upstream-Status / Applies-To / Maintainer / Last-Update / Signed-off-by
Upstream-PR (Submitted/Accepted/Backport) / Upstream-Commit (Accepted/Backport) / Whitelist-Reason (Rejected/Inappropriate/...)
```

### P13: PR 流程 + CI

**贡献者流程**：

```
1. cp my.patch → src/Redis-7.0.15/kunpeng-dtoe/
2. 写 DEP-3 头（From/Subject/Description/Origin/Upstream-Status/Applies-To/Maintainer/Last-Update/Signed-off-by）
3. bash tools/verify.sh     # lint + apply，一条命令
4. git commit + push → 开 PR
5. CI 自动跑 lint → Review → Merge
```

**CI 流程**：

```yaml
# .github/workflows/ci.yml — 1 step
name: ci
jobs:
  ci:
    steps:
      - uses: actions/checkout@v4
      - run: pip install pyyaml
      - run: SKIP_INSTALL=1 bash tools/verify.sh
```

## P14-P15：下一步计划

### P14: 仓库整改思路

**分三批推进**：

| 批次 | 仓数 | 代表 | 改造内容 | 优先级 |
|------|:--:|------|------|:--:|
| 第一批 | 3 | Redis, Kbox-patches, rocksdb | 完整 v6.0：manifest + DEP-3 + feature 目录 + tools + CI | 高 |
| 第二批 | ~15 | tensorflow, vllm-ops, zlib, lz4, protobuf, faiss... | 轻量 v6.0：manifest（3 行）+ DEP-3 头 | 中 |
| 第三批 | ~14 | rapidjson, snappy, hadoop, 3FS, ceph_BK... | 最简：manifest（3 行），DEP-3 头可选 | 低 |

**单个仓库改造步骤**：

```
1. 平铺 patch → 按 feature 分目录（同名序号 = 互斥 → 分目录 + conflicts）
2. 添加 manifest.yaml（3 行 + depends/conflicts 按需）
3. 添加 DEP-3 头到每个 .patch（From/Subject/Description/Origin/Upstream-Status/Applies-To/Maintainer/Last-Update/Signed-off-by）
4. 添加 tools/ + .github/workflows/（从 v6.0 模板复制）
5. bash tools/verify.sh → 全部通过 → 提交
```

**不改造的仓（63 个）**：完整源码仓、SIG 文档仓、工具/SDK 仓保持原样——不是 patch overlay，不需要 patch governance。

### P15: 整改 Skill 设计

```
opencode skill: boostkit-v6-migration

功能：
  1. 自动检测仓库类型（补丁 overlay / 源码 / 文档）
  2. 对于补丁仓：
     - 扫描 .patch 文件 → 发现 feature 分组（同名序号 = 冲突特征）
     - 生成 manifest.yaml（repo/version/commit 自动从现有 README 提取）
     - 生成 DEP-3 头模板（字段填空，维护人从 git log 提取）
     - 移动 patch 到 feature 子目录
     - 复制 tools/ + .github/ 模板
     - 运行 lint.py all → 报告缺失项
  3. 输出整改检查清单 + 未完成项

输入：gitcode.com/boostkit/<repo>.git URL
输出：整改后的仓库结构 + lint 报告 + 人工确认项清单
```
